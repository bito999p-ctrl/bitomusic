AetherMaster 技術仕様 & DSP分析

Web Audio API 信号フロー、実装されたアルゴリズム、および非採用技術に関する解説

AetherMaster は、最新のブラウザ標準である Web Audio API を用いて、クライアントサイド（ブラウザ内）だけで極めて低遅延・高音質なマスタリングを実行する仮想ハードウェア/DSPスイートです。

サーバーに音声を送ることなく、ローカルのCPUとブラウザのオーディオスレッドを使用してリアルタイムに処理を実行します。以下に、本ツールが採用している音声信号パス（Signal Chain）と技術仕様の詳細を解説します。

DSP シグナルパス (信号フロー)

音声信号は、デコードされた `AudioBuffer` から再生（またはオフラインレンダリング）される際、以下の順序で直列・並列に接続された各DSPノードを通過します。

セクション 使用Web Audio APIノード 処理概要と目的
1. Input Gain GainNode 入力音源の音量を事前にゲイン調整し、マスタリングチェーンに入る前段のヘッドルームを確保します。
2. Noise Cleaner (Rumble & Hiss) BiquadFilterNode × 2, WaveShaperNode (整流器), GainNode (変調量) 定常的な背景雑音を除去するノイズクリーニング段。
  • Rumble Cut (HPF)：80Hz以下の超低域空調ノイズや電源ハム音を急峻に物理遮断。
  • Hiss Reducer (動的VCF)：4kHz以上の高域エネルギーを監視するサイドチェーン検知器（HPF＋絶対値整流器＋10Hz平滑フィルター）がメインラインのLPFカットオフ周波数をダイナミックに変調（0%〜100%可変）。無音部分では4kHz付近まで閉じてヒスノイズを消し去り、演奏時は20kHzまで全開にして艶を100%保護します。
3. Parallel Saturator WaveShaperNode (並列), GainNode (Dry/Wet) 倍音を付加する飽和器。`WaveShaperNode`の伝達関数（カスタムシェイピングカーブ）を用いて、偶数・奇数倍音を生成。アナログテープ（Tape）や真空管（Tube）の質感をDry音にブレンドします。フィルター遅延による位相干渉を防ぐため、オーバーサンプリングは無効 (`none`) に設計されています。
4. 3-Band Equalizer BiquadFilterNode × 3 `lowshelf` (低域 EQ)、`peaking` (中域 EQ, 可変 Q: 0.3〜5.0)、`highshelf` (高域 EQ) の3つのフィルターを直列に接続。ジャンル特性に応じた音楽的な周波数のブースト/カットに加え、中域のQ値（帯域幅）を調整可能です。
5. AI Corrective Notches BiquadFilterNode × 8 (直列) AI が検出した高音域の鋭い不快ピーク（耳障りなレゾナンス音）をピンポイントで除去する、鋭いQ値（Q=12.0）の直列ノッチフィルター。Suno AI 特有の `9kHz〜10kHz` 付近のハーシュネスを優先的に検出してカットします。 (v3.21.4以降)：HISS REDUCERの動作量とリアルタイムに自動連動し、ノイズ低減が多い曲ではノッチフィルターのカット深さを最大65%自動緩和（ダッキング）し、高音の過剰カットや曇りを完全に防止します。
6. Glue Compressor DynamicsCompressorNode ゆるやかなコンプレッションによりダイナミクスを均一化。ソフトアタック（30ms前後）によりトランジェント（打楽器のアタック感）を残しつつ、全体をまとめ上げ（Glue）ます。
7. M/S Stereo Imager ChannelSplitterNode, ChannelMergerNode, GainNode (Mid/Side), BiquadFilterNode (Side HPF) 音源のL/Rを Mid（センター）と Side（広がり）にマトリックス変換して独立処理。
  • Mid：センター定位のボーカル・キックの輪郭を維持するため、ゲインは完全に `1.0` に固定。
  • Side：`width` スライダーで広がり幅を調整。さらに、低域の左右の位相干渉による低域のボヤけ（シュワシュワ音）を防ぐため、Side成分をカットする可変ハイパスフィルター（90Hz〜150Hz）を搭載。EDMやHipHopなどのクラブ系では150Hz、生楽器系では90Hzなど、ジャンル別で自動的に最適化されます。これにより低域はモノラル付近にタイトに保たれつつ、豊かな空間表現を可能にしています。
8. Maximizer Pre-Gain GainNode Brickwall Limiter に対してどれだけ突っ込むか（音圧の押し上げ）をデシベル（`limiterBoost`）単位で調整する増幅段です。
9. Brickwall Limiter DynamicsCompressorNode マスタリングの最終段で音圧を押し出すリミッター。`threshold=0dB`、応答を滑らかにするため `knee=3.0dB`、`ratio=20.0:1` に設定し、超過エネルギーを滑らかに圧縮。アタックタイムは最速の `0.1ms`、リリースタイムは低域の歪みを防ぐ最適な `80ms` に設計されています。
10. Safety Soft-Clipper WaveShaperNode 最終段で発生するデジタルクリッピング（ビビリ音）を完全に防止する安全装置。2倍オーバーサンプリングされたWaveShaperを使用し、振幅 `0.85` (-1.4dB) までは完全直線（透明）、それを超える瞬間的ピークのみを三次関数で滑らかにサチュレーションさせます。
11. Ceiling Gain GainNode デジタル配信サービス（Spotify、YouTube 等）にアップロードした際、非可逆圧縮（AAC/MP3）への変換時に生じるクリッピングを回避するため、最終出力の上限（天井）を `-1.0dB` 前後に制御してマスタリング処理を完了します。

テープ・チューブサチュレーションの数式シミュレーション

サチュレーターで使用されている `WaveShaper` 用の伝達関数（カーブ計算）は、以下の代数式を用いて対称・非対称の非線形歪みをリアルタイムに生成しています。

// Tape モード (Math.tanh によるソフトクリッピング。滑らかな倍音) f(x) = tanh(k * x) / tanh(k) [k = 0.5 + (Drive / 100) * 5.5] // Tube モード (非対称設計により偶数倍音を強調。温かく太い音) f(x) = (x < 0) ? (tanh(k * x) / tanh(k)) : (0.8 * x + 0.2 * x^2)

本格的だけど「あえて」やっていないこと（非採用技術）

プロ仕様のマスタリング機材やDAWプラグインでは定番とされていますが、AetherMaster においてクライアント動作の安定性と音質維持のために意図的に採用を見送った機能と、その技術的背景を公開します。

1. 先読みリミッター (Look-Ahead Limiter) の非採用

Look-Ahead とは：数ミリ秒（例：2〜5ms）先の信号を事前に読み取ることで、突発的なピーク（アタック）がスレッショルドを越える前にリミッターのゲインリダクションを立ち上げ、完全に「ノークリップ（超過ゼロ）」にする技術です。

非採用の理由と解決策：

  • Web Audio API の標準 `DynamicsCompressorNode` はリアルタイム処理用であり、Look-Ahead 時間を動的に設定するパラメータを持ちません。
  • これを手動（`AudioWorklet` など）で再現しようとすると、JSメインスレッドと音声スレッド間のバッファ遅延が発生し、リアルタイムでのスムーズなスライダー追従やスマホ動作時のCPU負荷（発熱）が許容値を超えてしまいます。
  • 代替アプローチ：AetherMaster では、リミッターの `attack` を限界まで速い `1ms` に設定し、かつ出力の最終段に `-1.0dB` の **Ceiling（天井）マージン** を設けることで、ルックアヘッド無しでもデジタルクリッピング（音割れ）を極めて効果的に防ぐアプローチをとっています。

2. マルチバンド・コンプレッサー (Multi-Band Compressor) の非採用

Multi-Band とは：音を「低域」「中域」「高域」などの周波数帯に分割し、それぞれの帯域に独立したコンプレッサーをかけてダイナミクスを制御する技術です。

非採用の理由と解決策：

  • Web Audio API 内で音域を分割する場合、複数の `BiquadFilterNode`（クロスオーバーフィルター）を使用しますが、Web Audio の基本フィルターはクロスオーバー領域で急峻な**位相の回転（Phase Distortion / コームフィルター現象）**を引き起こします。これにより、マスタリング後の音が全体的に痩せたり、特定の周波数が打ち消し合って音が不自然になるリスクがあります。
  • 高精度な Linkwitz-Riley フィルターを JS / Web Assembly で実装する手法もありますが、特に非力なスマートフォン端末において劇的な負荷上昇と発熱を招きます。
  • 代替アプローチ：単一の優れた **Glue Compressor**（シングルのまとまり重視）を使用し、その前段で **3バンドEQ** および **AIの8連 notch フィルター** であらかじめ問題となる共鳴帯域やバランスを高度に整えておくことで、マルチバンドを使用しなくても濁りのないクリアな音圧を実現しています。

3. ディザリング (Dithering / Noise Shaping) の非採用

Dithering とは：マスタリングデータの最終書き出し時（例：32bit float から 16bit integer に落とす時）に、極めて微小なホワイトノイズ（ディザー）を意図的に加え、デジタル量子化歪みを人間の耳に聞こえないノイズでマスキングする処理です。

非採用の理由と解決策：

  • AetherMaster からエクスポートされる 16bit WAV ファイル（`wav-exporter.js` 内で変換）は、浮動小数点数から `0x7FFF / 0x8000` への丸め処理（単純切り捨て・四捨五入）を行っています。
  • 現代のリスニング環境（特にユーザーの多くが Suno AI 等の生成音源をベースにしている点）や、最終配布先となるストリーミング配信サイト（Spotify、YouTube、TikTok 等）では、アップロード後に音源が必ず AAC/MP3 などの**非可逆圧縮形式（Lossy）に再エンコード**されます。
  • 非可逆圧縮コーデックのプロセスにおいては、ディザリングで加えた超微小ノイズは「不要な高域成分」として圧縮アルゴリズムにより真っ先に消去されるため、ディザリングの効果は理論上ほぼ完全に無効化されます。
  • そのため、書き出し処理の軽量化とコードサイズ削減を優先し、複雑なノイズシェイピング・ディザリングはあえて非搭載としています。
